## 1. Pfleger-Verwaltung

> Erstellt User Stories für die Verwaltung von Pflegekräften: mindestens Anlegen, Anzeigen, Ändern, Löschen.

### User Story 1: Pfleger Registrierung

**User Story**

> Als _Administrator_ möchte ich _Pflegekräfte im System anlegen_, um _sie für die Dienstplanung verfügbar_ zu machen.

**Akzeptanzkriterien**

| Nr. | Kriterium                                                                      |
|-----|--------------------------------------------------------------------------------|
| A_1 | Es gibt Pflichtfelder für Vorname, Nachname, Personalnummer, Qualifikation und Passwort. |
| A_2 | Die Personalnummer muss eindeutig sein.                                        |
| A_3 | Nach dem Speichern erscheint die neue Pflegekraft in der Übersicht.         |
| A_4 | Das Passwort wird verschlüsselt in der Datenbank gespeichert.         |

**Tasks**

| Nr. | Task                                                           |
|-----|----------------------------------------------------------------|
| T_1 | Modelklasse für Pflegekraft erstellen.                                   |
| T_2 | Datenbanktabelle für Pflegekräfte anlegen.              |
| T_3 | DAO für Pflegekräfte implementieren. |
| T_4 | Formular zum Anlegen einer Pflegekraft erstellen. |
| T_5 | Validierung für Pflichtfelder und Eindeutigkeit der Personalnummer implementieren. |
| T_6 | Passwort-Hashing beim Anlegen einer Pflegekraft integrieren. |

**Testfälle**

**TF1:**

| |                                                                         |
|---|-------------------------------------------------------------------------|
| **Vorbedingung** | Admin ist eingeloggt und befindet sich in der Pfleger-Verwaltung.        |
| **Testschritte** | 1. Button „Neu anlegen" klicken. 2. Gültige Daten in alle Felder eingeben. 3. Speichern klicken. |
| **Erwartetes Ergebnis** | Die neue Pflegekraft erscheint in der Übersichtsliste.                                         |

**TF2:**

| |                                                      |
|---|------------------------------------------------------|
| **Vorbedingung** | Admin befindet sich im Formular zum Anlegen einer Pflegekraft.                               |
| **Testschritte** | 1. Pflichtfelder leer lassen. 2. Speichern klicken.        |
| **Erwartetes Ergebnis** | Eine Fehlermeldung erscheint. Die Pflegekraft wird nicht gespeichert. |


### User Story 2: Pfleger Anzeigen

**User Story**

> Als _Pflegekraft_ möchte ich _alle Pflegekräfte in einer tabellarischen Übersicht sehen_, um _einen Überblick über das verfügbare Personal zu erhalten.

**Akzeptanzkriterien**

| Nr. | Kriterium                                                     |
|-----|---------------------------------------------------------------|
| A_1 | Alle Pflegekräfte werden mit Vorname, Nachname, Personalnummer und Qualifikation angezeigt.         |
| A_2 | Die Übersicht wird beim Öffnen der Pfleger-Verwaltung automatisch geladen.                       |
| A_3 | Neu angelegte oder geänderte Pflegekräfte werden sofort in der Übersicht angezeigt. |

**Tasks**

| Nr. | Task                                                           |
|-----|----------------------------------------------------------------|
| T_1 | Übersichts-View für Pflegekräfte erstellen.                                   |
| T_2 | Controller für die Pflegekräfte-Übersicht erstellen.              |
| T_3 | readAll()-Methode im PflegekraftDao implementieren. |

**Testfälle**

**TF1:**

| |                                                                         |
|---|-------------------------------------------------------------------------|
| **Vorbedingung** | Mindestens eine Pflegekraft ist in der Datenbank gespeichert.        |
| **Testschritte** | 1. Pfleger-Verwaltung im Hauptmenü öffnen. |
| **Erwartetes Ergebnis** | Alle gespeicherten Pflegekräfte werden mit vollständigen Daten (Vorname, Nachname, Personalnummer, Qualifikation) in der Tabelle angezeigt.                                         |


### User Story 3: Pfleger Ändern

**User Story**

> Als _Administrator_ möchte ich _die Daten einer Pflegekraft bearbeiten_, um _Fehler zu korrigieren oder aktuelle Informationen zu pflegen_.

**Akzeptanzkriterien**

| Nr. | Kriterium                                                                                |
|-----|------------------------------------------------------------------------------------------|
| A_1 | Alle Felder (Vorname, ,Nchname, Personalnummer, Qualifikation) konnen bearbeiten werden. |
| A_2 | Die geänderten Daten werden in der Datenbank gespeichert.                                |
| A_3 | Die Personalnummer bleibet auch nach der Änderung eindeutig.                             ||

**Tasks**

| Nr. | Task                                                 |
|-----|------------------------------------------------------|
| T_1 | Bearbeitungsfunktion in der Übersicht implemetieren. |
| T_2 | update() Methode im PflegekraftDao implementieren.   |
| T_3 | Validierung bei Änderung der Personalnummer implementieren.                                    |

**Testfälle**

**TF1:**

| |                                                                                                     |
|---|-----------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Mindestens eine Pflegekraft ist in der Übersicht sichtbar.                                          |
| **Testschritte** | 1. Einen Nachnamen per Doppelklick auswählen. 2. Den Namen ändern. 3. Eingabe mit Enter bestätigen. |
| **Erwartetes Ergebnis** | Der geänderte Name wird in der Übersicht angezeiget und ist in der Datenbank gespeichert.           |

**TF2:**

| |                                                                                                                 |
|---|-----------------------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Zwei Pflegekräfte mit unterschiedlichen Personalnummer existieren.                                              |
| **Testschritte** | 1. Personalnummer einer Pflegekraft auf die bereits vergebene Nummer der anderen ändern. 2. Eingabe bestätigen. |
| **Erwartetes Ergebnis** | Eine Fehlermeldung erscheint. Änderung wird nicht gespeichert.                                                                               |

### User Story 4: Löschen

**Akzeptanzkriterien**

| Nr. | Kriterium                                                                  |
|-----|----------------------------------------------------------------------------|

**Tasks**

| Nr. | Task                                               |
|-----|----------------------------------------------------|

**Testfälle**

**TF1:**

| |                                                                                                             |
|---|-------------------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Mindestens eine Pflegekraft ist in der Übersicht sichtbar.                                                  |
| **Testschritte** | 1. Eine Pflegekraft in der Tabelle auswählen. 2. Löschen-Button klicken. 3. Bestätigung im Dialog klicken.. |
| **Erwartetes Ergebnis** | Die Pflegekraft wird aus der Übersicht entfernt und in der Datenbank gelöscht.                             |

**TF2:**

| |                                                                                                             |
|---|-------------------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Mindestens eine Pflegekraft ist in der Übersicht sichtbar.                                                  |
| **Testschritte** | 1. Eine Pflegekraft auswählen. 2. Löschen-Button klicken. 3. Im Bestätigungsdialog auf „Abbrechen" klicken. |
| **Erwartetes Ergebnis** | Die Pflegekraft bleibt in der Übersicht und in der Datenbank.                                                |

### User Story 5: Passwort ändern

**User Story**

> Als _eingeloggte Pflegekraft_ möchte ich _mein Passwort ändern_, um _meine Zugangsdaten aktuell und sicher zu halten_.

**Akzeptanzkriterien**

| Nr. | Kriterium                                                                  |
|-----|----------------------------------------------------------------------------|
| A_1 | Die eingeloggte Pflegekraft kann ihr eigenes Passwort ändern.                         |
| A_2 | Zur Bestätigung muss das aktuelle Passwort eingegeben werden.                        |
| A_3 | Das neue Passwort wird verschlüsselt in der Datenbank gespeichert. |
| A_4 | Eine Pflegekraft kann ausschließlich ihr eigenes Passwort ändern, nicht das anderer. |

**Tasks**

| Nr. | Task                                               |
|-----|----------------------------------------------------|
| T_1 | Passwort-ändern-Formular in der Anwendung erstellen.        |
| T_2 | Verifizierung des aktuellen Passworts vor der Änderung implementieren.                 |
| T_3 | update()-Methode im PflegekraftDao für das Passwort-Feld erweitern.. |

**Testfälle**

**TF1:**

| |                                                                                                             |
|---|-------------------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Pflegekraft ist eingeloggt.                                               |
| **Testschritte** | 1. Passwort-ändern-Formular öffnen. 2. Aktuelles Passwort korrekt eingeben. 3. Neues Passwort eingeben und bestätigen. 4. Speichern klicken. |
| **Erwartetes Ergebnis** | Das Passwort wurde geändert. Bei erneutem Login ist nur das neue Passwort gültig                             |

**TF2:**

| |                                                                                                             |
|---|-------------------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Pflegekraft ist eingeloggt.                                               |
| **Testschritte** | 1. Passwort-ändern-Formular öffnen. 2. Falsches aktuelles Passwort eingeben. 3. Speichern klicken. |
| **Erwartetes Ergebnis** | Eine Fehlermeldung erscheint. Das Passwort wird nicht geändert.                                               |

## 2. Datenschutz-Fixes

### Priorisierungs-Begründung

> Warum habt ihr gerade diese drei Verstöße priorisiert (Schadensrisiko, Aufwand, rechtliches Gewicht)? **Eine der drei Stories muss die Zugangsbeschränkung behandeln (Login-System).**

_Das Pflegemodul sowie die implementierung des Login-Systems sind vorgaben innerhalb des Projektes, aber auch enorm wichtig für die rechtskonforme Umsetzung der Programmintention._
_Das Exportmodul ermöglicht Patienten, Zugriff auf Ihre personenbezogenen Daten zu erhalten._
_Die Rechteverwaltung reguliert unbefugte Zugriffe und ermöglicht unterschiedliche Sichtbarkeiten basierend auf der zugewiesenen Rolle. Elementar wichtig, um DSGVO-Konform zu agieren._

### User Story 1: Login System

**User Story**

> Als _authorisierter Nutzer_ möchte ich _mich mit Benutzernamen und Password authentifizieren_, um _sicherzustellen, dass nur berechtigte Personen Zugriffe auf Patientendaten haben._

**Akzeptanzkriterien**

| Nr. | Kriterium                                                                            |
|-----|--------------------------------------------------------------------------------------|
| A_1 | Zugriff auf das Programm ist nur nach Login möglich                                  |
| A_2 | Passwörter müssen verschlüsselt gespeichert werden                                   |
| A_3 | Bei falschem Benutzernamen oder Passwort wird eine generische Fehlermeldung gegeben. |
| A_4 | Nach X fehlgeschlagenen Versuchen wird der Account temporär gesperrt.                |
| A_5 | Eine aktive Session läuft nach X Minuten Inaktivität automatisch ab                  |
| A_6 | Der Nutzher kann sich ausloggen wobei die Session invalidiert wird                   |

**Tasks**

| Nr. | Task                                                                                                              |
|-----|-------------------------------------------------------------------------------------------------------------------|
| T_1 | User Datenbankschema implementieren (id, username, password_hash, role, failed_attempts, locked_until, created_at) |
| T_2 | Passwort-Hashing mit jbcrypt integrieren                                                                          |
| T_3 | Login-Logik, Fehlerbehandlung, Account-Sperre                                       |
| T_4 | Eingeloggten User im Speicher behalten, Session Timeout                           |
| T_5 | Event-Filter für den Session Timeout einbauen                                                                     |
| T_6 | Frontend Login View und Form bauen                                                                                |
| T_7 | Logout Funktion implementieren                                                                                    |

**Testfälle**

**TF1:** Zugriff ohne Login wird gesperrt

| |                                                                                 |
|---|---------------------------------------------------------------------------------|
| **Vorbedingung** | User ist nicht eingeloggt                                                       |
| **Testschritte** | 1. Versuchen die Pflegerliste zuöffnen 2. Versuchen die Patientenliste zuöffnen |
| **Erwartetes Ergebnis** | System leitet automatisch zum Login weiter wenn der User nicht eingeloggt ist.  |

**TF2:** Erfolgreicher Login

| |                                                                          |
|---|--------------------------------------------------------------------------|
| **Vorbedingung** | User existiert in der SQLite-DB                                          |
| **Testschritte** | 1. Korrekten Username und Passwort eingeben 2. Login Button klicken      |
| **Erwartetes Ergebnis** | User wird im `SessionManager` gespeichert, App wechselt zur Hauptansicht |

**TF3:** Fehlgeschlagener Login

| |                                                                              |
|---|------------------------------------------------------------------------------|
| **Vorbedingung** | User existiert in der SQLite-DB                                              |
| **Testschritte** | 1. Korrekten Username und falsches Passwort eingeben 2. Login Button klicken |
| **Erwartetes Ergebnis** | Fehlermeldung wird angezeigt, `failed_attempts` wird in der DB um 1 erhöht   |


**TF4:** Account-Sperre nach N fehlgeschlagenen Versuchen

| |                                                                                                            |
|---|------------------------------------------------------------------------------------------------------------|
| **Vorbedingung** | User hat bereits N-1 fehlgeschlagene Versuche in der DB                                                    |
| **Testschritte** | 1. Erneut falsches Passwort eingeben 2. Login Button klicken                                               |
| **Erwartetes Ergebnis** | `locked_until` wird in DB gesetzt, Fehlermeldung mit Sperrdauer erscheint, Neustart umgeht die Sperre nicht |


**TF5:** Session Timeout

| |                                                                                            |
|---|--------------------------------------------------------------------------------------------|
| **Vorbedingung** | User ist eingeloggt                                                                        |
| **Testschritte** | 1. X Minuten keine Maus oder Tastatur aktivität 2. Einen Button klicken oder View wechseln |
| **Erwartetes Ergebnis** | 'SessionManager' loggt den User aus, Login View wird angezeigt                             |



**TF6:** Logout

| |                                                           |
|---|-----------------------------------------------------------|
| **Vorbedingung** | Use ist eingeloggt                                        |
| **Testschritte** | 1. Logout-Button klicken                                  |
| **Erwartetes Ergebnis** | `SessionManager` wird geleert, App wechselt zur Login View |

## 3. Datenverwaltung


### User Story 1: Export-System

**User Story**

> Als _authorisierter Mitarbeiter_ möchte ich _Patientendaten DSGVO-Konform exportieren können_, um _dem Recht auf Auskunft gerecht zu werden_.

**Akzeptanzkriterien**

| Nr. | Kriterium                                                                                                                                                                                           |
|-----|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| A_1 | Ein Button „Export“ ist ausschließlich für Benutzer mit der Rolle „Verwaltung“ und "Admin" sichtbar.                                                                                                |
| A_2 | Nach einem Klick auf den Button „Export“ und der Auswahl des Patienten werden automatisch zwei Dateien im JSON- und PDF-Format erstellt und in einem definierten Verzeichnis gespeichert.           |
| A_3 | Die PDF-Datei enthält eine strukturierte Darstellung der personenbezogenen Daten des Patienten sowie dessen Behandlungsdaten. Daten von Pflegern oder anderen Mitarbeitern werden nicht exportiert. |


**Tasks**

| Nr. | Task                                                                         |
|-----|------------------------------------------------------------------------------|
| T_1 | PDF Export-Service erstellen                                                 |
| T_2 | Json Export-Service erstellen                                                |
| T_3 | Implementierung des Buttons abhängig von Rolle                               |
| T_4 | Implementierung der Patientenauswahl                                         |
| T_5 | Abfrage der Patienten- und Behandlungsdaten                                  |
| T_6 | Export der Patienten- und Behandlungsdaten unter Ausschluss der Pflegerdaten |
| T_7 | Fehlerbehandlung                                                             |

**Testfälle**

**TF1:** Sichtbarkeit des Buttons als "Verwaltung" oder "Admin"

| |                                                              |
|---|--------------------------------------------------------------|
| **Vorbedingung** | User ist mit der Rolle "Verwaltung" oder "Admin" eingeloggt. |
| **Testschritte** | 1. Öffnen der Patientenübersicht                             |
| **Erwartetes Ergebnis** | Button "Export" wird angezeigt.                              |


**TF2:** Sichtbarkeit des Buttons ohne berechtigung

| |                                                                      |
|---|----------------------------------------------------------------------|
| **Vorbedingung** | User ist *nicht* mit der Rolle "Verwaltung" oder "Admin" eingeloggt. |
| **Testschritte** | 1. Öffnen der Patientenübersicht                                     |
| **Erwartetes Ergebnis** | Button "Export" wird nicht angezeigt und kann nicht angeklickt werden.|

**TF3:** Patientenauswahl

| |                                                                                   |
|---|-----------------------------------------------------------------------------------|
| **Vorbedingung** | User ist mit der Rolle "Verwaltung" oder "Admin" eingeloggt.                      |
| **Testschritte** | 1. Öffnen der Patientenübersicht  2. Klick auf "Export" 3. Auswahl des Patienten  |
| **Erwartetes Ergebnis** | Eine Auswahl der verfügbaren Patienten, für die Daten vorhanden sind, taucht auf. |

**TF4:** Erstellung der Export Dateien

| |                                                                                                     |
|---|-----------------------------------------------------------------------------------------------------|
| **Vorbedingung** | Patientenauswahl wurde geöfnet.                                                                     |
| **Testschritte** | 1. Auswahl eines Patienten 2. Klick auf "Export"                                                    |
| **Erwartetes Ergebnis** | Eine PDF- und eine Json-Datei werden mit dem erwarteten Inhalt im angegebenen Verzeichnis erstellt. |
